# 📜 Test Coding Skill: 黄金のテスト原則

このスキルは、バックエンドおよびフロントエンドのテストにおいて、「生きた仕様書（Living Documentation）」となる、意味のあるテストコードを生成するためのガイドラインです。

## 📍 1. 構造の原則 (AAA Pattern)

すべての `it` ブロックは、以下の 3 つのセクションで構成し、明示的なコメントを付与してください。

1.  **Arrange**: データのモック、環境変数の設定、前提条件の構築。
2.  **Act**: テスト対象の関数やエンドポイントの呼び出し。
3.  **Assert**: 期待される結果（ステータスコード、レスポンスボディ、型の整合性）のチェック。

```typescript
it('テスト名', async () => {
  // ## Arrange ##
  const mockEnv = { ... };

  // ## Act ##
  const res = await target.execute();

  // ## Assert ##
  expect(res).toBe(...);
});
```

## 📍 2. 網羅性の原則 (Matrix)

以下の 3 つのケースを必ず検討してテストを作成してください。

### ① 正常系 (Normal Cases)
- **目的**: 基本機能が意図通りに動作することを証明する。
- **検証項目**: ステータス 200/201、レスポンスの型の正しさ（Zod Schema）、値の正確性。

### ② 異常系 (Abnormal Cases)
- **目的**: 誤った入力や外部エラー時に「安全に失敗」することを証明する。
- **検証項目**: ステータス 400（バリデーション失敗）、500（サーバーエラー）、エラーメッセージの明瞭さ。

### ③ 境界値 (Boundary Cases)
- **目的**: 動作の限界（エッジケース）での振る舞いを証明する。
- **検証項目**: `limit=0`、空のリスト、極端に長い文字列、最大/最小値の価格など。

## 📍 3. ドキュメンテーションの原則

テストコードは、**「コードを読めばその機能の使い方がわかる」**ように記述してください。

- **明瞭な説明**: `it('should return ...')` だけでなく、日本語で「〜が〜である時、〜を返すこと」など、ビジネスルールに基づいた名称にする。
- **意味のあるモック**: 意味のないダミーデータではなく、実データに近い形式をモックとして使用する。
