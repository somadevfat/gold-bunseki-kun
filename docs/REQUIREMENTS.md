# 要件定義およびアーキテクチャ: Gold Volatility Analysis (Gold-Vola-Bunseki)

## 1. 目的
XAUUSD（ゴールド）に特化し、経済指標が生じさせるボラティリティ（平均値）と、現在の市場のボラティリティ（地合い）を比較・分析する。これにより、現在の相場が統計的に見てどれほど逸脱しているかを可視化する。

## 2. ターゲット
- **通貨ペア**: XAUUSD (Gold/USD)
- **データ期間**: 過去5年間 + 直近1ヶ月（最大で10年以上前まで遡行可能）

## 3. 機能要件

### 3.1 データ連携・インポート (外部API連携)
- **データソース**: Twelve Data API（商用利用: Growプラン等を想定）
- **取得データ**: XAUUSDの1分足ヒストリカルデータ、およびリアルタイム（または定期更新）データ。
  - *※過去のMT5からの手動エクスポート処理は廃止し、完全自動のデータ収集パイプラインを構築する。*
- **経済指標データ**: 各種経済指標の発表時刻や結果データを自動で取得し、タイムスタンプベースで価格データと紐付ける。

### 3.2 ボラティリティ算出ロジック
- **時間帯の定義**: 以下の時間枠でボラティリティを区切る。
  - 07:00 - 09:00
  - 09:00 - 12:00
  - 12:00 - 16:00
  - 16:00 - 21:00
  - 21:00 - 24:00
  - 00:00 - 03:00
  - 03:00 - 06:00
- **算出方法**:
  - 各時間枠内における価格の高値と安値の差分（ボラティリティ）を計算（ZigZag的な挙動をパース）。
  - その時間枠内に発生した経済指標を、そのボラティリティの要因として紐付ける。
  - 同じ指標が発生した際の結果を蓄積し、指標ごとの平均ボラティリティを統計データとして算出する。

### 3.3 地合い分析（ボラティリティ比較）
- **基準値設定**: 解析された統計データに基づき、時間帯ごとの「標準的なボラティリティ」を設定する（UIやAPIから容易に変更可能にすること）。
- **逸脱度判定**: 現在のボラティリティが基準値から何ドル外れているかによって、「大・小」などの偏差判定を行い、フロントエンドに描画する。

## 4. アーキテクチャ構成と使用技術

ポートフォリオとして実務水準（商用環境）の要件を満たすため、保守性に優れた適材適所のマイクロサービスアーキテクチャを採用する。
**内部設計にはクリーンアーキテクチャを採用し、ビジネスロジックと外部依存（技術・インフラ等）を明確に分離することで、高度なテスト容易性と保守性を証明する。**

### 4.1 全体技術スタック
- **フロントエンド**: Next.js (App Router) + TailwindCSS
  - ランタイム/パッケージ管理: **Bun** (Node.js不使用)
  - **選定理由**: モダンReactのデファクトスタンダードであり、SSR/RSCによるパフォーマンス最適化の知見を証明できるため。また、最速のランタイムであるBunを使用し、最新エコシステムへのキャッチアップ力をアピールする。
- **バックエンド (メインAPI)**: Go (EchoまたはGin)
  - **選定理由**: API通信における極めて高いパフォーマンスと低メモリ消費が魅力。クリーンアーキテクチャを適用し、HTTPハンドラ、ユースケース、リポジトリ層を分離して構築する。
- **分析・統計計算**: Python (FastAPI + Pandas / NumPy等)
  - **選定理由**: 重たい数値計算ロジックとメインAPIを分離するため。FastAPIなどのインフラ層から切り離された純粋な計算処理として実装し、堅牢な独立性を確保する。
- **コード品質・テスト**: Vitest (フロント), pytest (Python), Go本体のテストツール (テスト駆動開発TDDを順守)

### 4.2 インフラ環境 (AWS)
単一のEC2ではなく、高負荷なボラティリティ分析とAPI通信に備えたスケーラブルなコンテナ運用設計。

- **コンピューティング**: AWS ECS on Fargate
  - フロント、API、分析サーバーをそれぞれDockerコンテナ化し、サーバーレスコンテナで運用。実務標準のオーケストレーション手法を適用。
- **データベース**: Amazon Aurora PostgreSQL (またはRDS)
  - APIから毎秒・毎分取得した時系列データやボラティリティの算出結果を保管する。
- **オブジェクトストレージ**: Amazon S3
  - 数年分の生データアーカイブ（CSV/JSON形式）を利用・保管するためのデータレイク。
- **バッチ処理**: Amazon EventBridge + AWS Lambda (またはECSタスク)
  - 毎晩自動でAPIを叩き、最新の価格・指標データを収集・DBに保存するジョブを回し、データ鮮度を維持する。

## 5. 面接用ポートフォリオとしてのアピールポイント（総括）
1. **適材適所の技術選定とクリーンアーキテクチャ**: 各層（Next.js / Go / Python）で技術の強みを活かすと共に、クリーンアーキテクチャに基づき関心の分離を徹底。テストが容易で変化に強い設計を実現。
2. **実運用に即したデータパイプライン**: ローカルでの手作業を完全に脱却し、AWS上で商用APIからの自動データ取込〜蓄積〜分析の一連の流れを完結させている。
3. **保守性とスケーラビリティの設計**: フルマネージドなコンテナ環境（ECS Fargate）を利用し、トラフィック増に対するスケール設計とマイクロサービス的な責務分割を論理的に説明可能。
